# 日报资讯采集指南

用于“生成 AI 日报”时的资讯采集阶段。只有在进入生成流程、且需要执行资讯采集时，才读取本文件。

目标：

- 优先获取最近 3 日内的 AI 资讯
- 先广泛建立候选池，再少量深挖正文
- 优先覆盖权威官方来源，并补充研究、社区和中文信号
- 在进入筛选和改写前，完整保留原始采集数据
- 优先使用 Agent Reach 作为资讯采集入口

---

## 一、采集总原则

1. 默认只采集最近 3 日内的资讯；如果用户明确指定更短时间范围，则以用户要求为准
2. 默认优先使用 Agent Reach 抓取；只有在 Agent Reach 当前不可用或明显不足以完成任务时，才退回到普通搜索 / fetch / reader 工具
3. 如果当前环境没有安装 Agent Reach 技能，应先完成安装，再进入采集
4. 不要一开始就抓取所有文章全文，先建立轻量候选池
5. 前 5 条候选优先从官方一手源中产生
6. 媒体、社区、研究、开源信号用于补足多样性，不应长期缺位
7. 采集过程必须完整留存原始数据，再进入去重、筛选、排序和改写

Agent Reach 安装要求：

- 安装文档：`https://raw.githubusercontent.com/Panniantong/agent-reach/main/docs/install.md`
- 若未安装，先按该文档完成安装和健康检查，再继续采集
- 安装完成后，优先使用 Agent Reach 已提供的渠道能力建立候选池和执行深抓

### Agent Reach 强制检查

进入采集阶段后，必须先完成以下动作，才能开始真正抓取：

1. 运行 `agent-reach doctor`
2. 记录 doctor 输出里的可用渠道、不可用渠道、缺失配置与代理要求
3. 根据 doctor 结果选择上游工具；不要把 Agent Reach 误解成单一抓取命令
4. 只有在确认对应渠道不可用、未配置、被权限限制、或目标来源无法覆盖时，才允许退回其他工具

这不是建议，而是执行门槛。没有完成这一步，不应直接进入普通搜索。

原始文本格式示例见：

- `reference/raw_capture_example.txt`

## 二、推荐采集方案

默认采用两阶段方案。

### 第一阶段：轻量候选池采集

先从频道页、列表页、搜索结果页、官方 News / Blog 索引页建立候选池，只获取轻量信息：

- 标题
- 简介 / 导语 / snippet
- 来源
- 链接
- 发布时间（如可获得）
- 频道 / 主题

优先入口：

- Agent Reach 可用渠道的搜索结果页、RSS、社区搜索
- 官方 News / Blog 列表页
- 媒体 AI 频道页 / 栏目页
- GitHub Trending / Release 列表页
- Hugging Face Blog / 社区列表页

第一阶段目标：

- 快速了解最近一周有哪些候选资讯
- 广覆盖、低成本地建立候选池
- 为后续深挖和筛选提供上下文

### 第二阶段：按需深挖正文

只有在第一轮筛选后，才对少量高价值条目继续抓全文或长正文。

优先深抓对象：

- 可能进入前 5 条的核心资讯
- 标题和简介不足以判断的重要条目
- 需要核实发布时间、关键事实、产品细节的条目
- AI 判断值得进一步深读的 5-10 条候选

第二阶段可读取：

- 文章正文
- 官方发布说明
- 系统卡 / 论文摘要
- GitHub README / Release Note

## 三、时间约束

默认时间窗口：

- 最近 3 日

强制要求：

- 例如当前日期若为 `2026-03-24`，默认有效窗口就是 `2026-03-22` 到 `2026-03-24`（即当天往前推 2 天）
- 超出这个窗口的内容不得作为本期资讯入选
- 如果需要引用更早内容，只能作为背景材料，且必须明确标注“背景”，不能混入本期条目
- 在生成最终 payload 前，必须对每一条入选资讯再做一次日期校验
- 只要发现超窗条目，必须替换，不能保留

执行要求：

- 优先保留最近 3 日内的内容
- 如果某个来源在最近 3 日没有更新，不要拿更旧内容硬凑
- 如果确实需要补充背景，可读取更早资料，但必须明确标注它是背景材料而非本期新资讯

采集阶段日期确认要求：

- 在采集每一条候选时，必须记录该条目的实际发布日期（精确到日）
- 搜索结果中如果只显示"X days ago"、"last week"等模糊时间，必须换算为具体日期再判断是否在窗口内
- 如果一条资讯无法确认发布日期，应在候选池中标注"日期不明"，筛选时降低其优先级
- 禁止在 `time_label` 中使用 `本周`、`持续热门`、`持续活跃` 等模糊表述，必须写具体日期（如 `3月22日`、`今天`）

## 四、来源策略

### 核心来源

优先从以下来源构建主候选池：

- OpenAI
- Anthropic
- Google DeepMind / Gemini
- Microsoft
- NVIDIA
- DeepSeek
- 通义 / Qwen
- 智谱
- 百度 / 文心

### 多样性补充

候选池中应尽量补齐以下类别：

- 官方产品 / 模型 / 平台更新
- 国内媒体或官方中文信号
- 开源 / GitHub / 社区信号
- 研究 / Benchmark / 论文 / 安全信号

GitHub 开源信号不是可选项，而是默认必查项：

- 每期至少检查一次 GitHub Trending 或 GitHub Search
- 如果最近 3 日内存在明显升温的 AI 项目，日报中至少纳入 1-2 条 GitHub 开源条目
- 如果没有纳入，必须在最终汇报中说明为什么

### 媒体源处理方式

媒体源优先从频道页建立候选池，再按需进入单篇文章正文。

例如：

- 36 氪优先从 `https://36kr.com/information/AI/` 建候选池

## 五、推荐工具使用方式

可以使用以下任一方式采集：

- Agent Reach 已可用渠道
- 搜索工具
- fetch / reader 工具
- `scripts/save_raw_capture.py`

推荐流程：

1. 先运行 `agent-reach doctor`
2. 根据 doctor 结果，优先用 Agent Reach 已打通的上游工具建立 `index` 候选池，例如：
   - X/Twitter：`bird`
   - GitHub：`gh`
   - 网页正文：`curl https://r.jina.ai/URL`
   - Exa / 微博 / 小红书 / 抖音 / LinkedIn：`mcporter`
   - RSS：RSS 读取器
3. 结合用户兴趣与候选质量做第一轮筛选
4. 对少量高价值条目再用 doctor 已确认可用的上游工具深抓正文
5. 将深抓结果保存为 `detail` 原始数据
6. 如果有任何来源改用备用工具，必须在原始采集记录或最终汇报里注明原因

GitHub 执行建议：

- 在当前技能仓库内，默认优先执行：
  - `python3 scripts/fetch_github_agent_trends.py --period monthly --limit 10 --json`
- 该脚本复用了 `github-agent-trends` 的实现方案：
  - topic 搜索：`topic:ai-agent`、`topic:agentic-ai`、`topic:autonomous-agents`、`topic:llm-agents`
  - 关键词搜索：`"AI agent"`、`"autonomous agent"`、`"coding agent"` in:name,description
  - 活跃度筛选：`pushed:>=since`
  - 增速排序：`daily_stars = stars / days_old`
- 如果脚本失败，再退回 GitHub Trending 的当日 / 近日热门仓库
- 再用 `gh search repos` 补充最近 3 日高活跃的 AI 仓库
- 重点记录：仓库名、定位、星数或热度信号、最近 push 时间、与你相关的原因

如果使用 Agent Reach：

- 优先使用 `agent-reach doctor` 已标记为可用的上游工具
- 社交媒体和社区渠道可作为补充，不要替代官方源
- 抓到的是搜索摘要时，可在第二阶段再用 reader 深抓正文
- 如果 `agent-reach doctor` 显示某些渠道不可用，应记录不可用情况，再决定是否退回普通工具
- 最终汇报必须明确写出：
  - 是否运行了 `agent-reach doctor`
  - 本次实际使用了哪些 Agent Reach 上游工具
  - 哪些渠道因 doctor 显示不可用而被跳过

## 六、原始数据留存要求

这是强制要求：所有拉取到的资讯原始数据都必须留存，再进入筛选和加工。

默认输出：

- 第一阶段：`output/raw/{date}_index.txt`
- 第二阶段：`output/raw/{date}_detail.txt`

### 第一阶段建议保留

- 抓取时间
- 搜索词、频道页任务或来源页任务
- 来源名 / 来源类型
- 原始标题
- 原始链接
- 发布时间（如可获得）
- 简介 / 导语 / snippet

### 第二阶段建议保留

- 抓取时间
- 深挖任务
- 来源名 / 来源类型
- 原始标题
- 原始链接
- 语言
- 清洗后的正文、页面片段或发布说明
- 若抓取受阻，保留真实返回内容并标记异常状态

### 文本格式要求

- 每条记录用明显分隔线分开
- 元信息写成键值行
- 后面紧跟抓取到的原始文本
- 允许持续追加，不要求固定 JSON 结构

---

## 七、采集约束

1. 不要跳过第一阶段直接全文抓取所有文章
2. 不要只保留最后入选日报的条目，必须保留全部候选池
3. 原始文件内容应尽量贴近抓取结果本身，不要提前改写成日报文案
4. 第二阶段只深挖少量高价值条目，不要默认把所有候选全文抓下来
5. 如果页面存在明显验证码、反爬、挑战页，也应保留真实返回结果，并在后续筛选时降低其可用性

---

## 八、最小执行清单

进入筛选与加工前，至少确认：

- 已先运行 `agent-reach doctor`
- 已建立最近 3 日候选池
- 已优先覆盖权威官方来源
- 已补充至少一部分国内或社区 / 研究信号
- 已检查 GitHub 热门 / 高活跃 AI 开源项目
- 已将候选池原始文本保存到 `index`
- 已对少量高价值条目完成 `detail` 深抓
- 已将深抓原始文本保存到 `detail`
- 如有退回普通工具，已记录退回原因与补齐来源
