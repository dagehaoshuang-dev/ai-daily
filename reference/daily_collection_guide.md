# 日报资讯采集指南

用于"生成日报"时的资讯采集阶段。只有在进入生成流程、且需要执行资讯采集时，才读取本文件。

目标：

- 优先获取用户关注领域内最近 3 日的资讯
- 先广泛建立候选池，再少量深挖正文
- 优先覆盖 profile.yaml 中配置的一手来源，并补充社区和多样性信号
- 在进入筛选和改写前，完整保留原始采集数据
- 优先使用 Agent Reach 作为资讯采集入口

---

## 一、采集总原则

1. 默认只采集最近 3 日内的资讯；如果用户明确指定更短时间范围，则以用户要求为准
2. **国内源优先**：候选池应以中文/国内来源为主体，海外来源仅在该领域确实重要时纳入（例如某个海外产品发布直接影响国内市场）
3. 默认优先使用 Agent Reach 抓取；只有在 Agent Reach 当前不可用或明显不足以完成任务时，才退回到普通搜索 / fetch / reader 工具
4. 如果当前环境没有安装 Agent Reach 技能，应先完成安装，再进入采集
5. 不要一开始就抓取所有文章全文，先建立轻量候选池
6. 前 5 条候选优先从 `sources.direct` 中配置的一手源产生
7. 社区、研究、开源等多样性信号用于补足，不应长期缺位
8. 采集过程必须完整留存原始数据，再进入去重、筛选、排序和改写

Agent Reach 安装要求：

- 安装文档：`https://raw.githubusercontent.com/Panniantong/agent-reach/main/docs/install.md`
- 若未安装，先按该文档完成安装和健康检查，再继续采集
- 安装完成后，优先使用 Agent Reach 已提供的渠道能力建立候选池和执行深抓

### Agent Reach 强制检查

进入采集阶段后，必须先完成以下动作，才能开始真正抓取：

1. 运行 `agent-reach doctor`
2. 记录 doctor 输出里的可用渠道、不可用渠道、缺失配置与代理要求
3. 根据 doctor 结果选择上游工具；不要把 Agent Reach 误解成单一抓取命令
4. 只有在确认对应渠道不可用、未配置、被权限限制、或目标来源无法覆盖时，才允许退回其他工具

这不是建议，而是执行门槛。没有完成这一步，不应直接进入普通搜索。

原始文本格式示例见：

- `reference/raw_capture_example.txt`

---

## 二、推荐采集方案

默认采用两阶段方案。

### 第一阶段：轻量候选池采集

先从频道页、列表页、搜索结果页、官方索引页建立候选池，只获取轻量信息：

- 标题
- 简介 / 导语 / snippet
- 来源
- 链接
- 发布时间（如可获得）
- 频道 / 主题

优先入口：

- Agent Reach 可用渠道的搜索结果页、RSS、社区搜索
- `sources.direct` 中配置的官方页面和频道页
- `sources.search_seeds` 中的搜索种子构造的搜索查询
- 如果 profile 中包含 GitHub 相关来源或搜索种子，检查 GitHub Trending / Search

第一阶段目标：

- 快速了解最近 3 日内有哪些候选资讯
- 广覆盖、低成本地建立候选池
- 为后续深挖和筛选提供上下文

### 第二阶段：按需深挖正文

只有在第一轮筛选后，才对少量高价值条目继续抓全文或长正文。

优先深抓对象：

- 可能进入前 5 条的核心资讯
- 标题和简介不足以判断的重要条目
- 需要核实发布时间、关键事实、产品细节的条目
- 值得进一步深读的 5-10 条候选

第二阶段可读取：

- 文章正文
- 官方发布说明
- 技术文档 / 论文摘要
- GitHub README / Release Note

---

## 三、时间约束

默认时间窗口：

- 最近 3 日

强制要求：

- 超出时间窗口的内容不得作为本期资讯入选
- **搜索查询本身应包含时间限定**（如在关键词后加"2026年3月22日-24日"、"site:xxx after:2026-03-22"等），从源头控制候选池时效性
- 候选池中每条资讯必须标注可验证的发布日期；无法确认日期的条目降低优先级或排除
- 如果需要引用更早内容，只能作为背景材料，且必须明确标注"背景"，不能混入本期条目
- 在生成最终 payload 前，必须对每一条入选资讯再做一次日期校验
- 只要发现超窗条目或日期模糊的条目，必须替换，不能保留

执行要求：

- 优先保留最近 3 日内的内容
- 如果某个来源在最近 3 日没有更新，不要拿更旧内容硬凑
- 不要用"本周"、"近日"等模糊标注代替实际发布日期
- 如果确实需要补充背景，可读取更早资料，但必须明确标注它是背景材料而非本期新资讯

---

## 四、来源策略

### 核心来源

优先从 `config/profile.yaml` 中 `sources.direct` 配置的来源构建主候选池。这些是用户领域内的一手权威来源。

如果 `sources.direct` 为空或来源不足：

1. 根据用户的 `topics` 和 `keywords` 动态构造搜索查询
2. 如果 `sources.search_seeds` 有内容，用这些种子词执行搜索
3. 如果 sources 整段为空，完全基于 topics/keywords 自行搜索，不要因此停止采集

### 国内源优先原则

本技能面向国内用户，采集时应遵循以下优先级：

1. **国内一手源**（官方公众号、官网公告、行业媒体）→ 最优先
2. **国内社区和讨论**（知乎、微信公众号、行业论坛）→ 补充信号
3. **海外权威源**（仅在该条目对国内用户确实重要时纳入）→ 有选择地引入

判断海外源是否值得纳入的标准：
- 该事件直接影响国内市场或用户（如政策变化、平台合作）
- 该领域的核心玩家本身就是海外公司（如支付领域的 Visa/Mastercard）
- 国内暂无对等报道，但信息价值很高

搜索查询应以中文关键词为主，英文关键词为辅。

### 多样性补充

候选池中应尽量补齐以下信号类别：

- 一手产品 / 平台 / 行业更新
- 国内媒体报道和聚合信号
- 社区讨论和用户反馈
- 开源 / GitHub 信号（如果用户领域涉及开源项目）
- 研究 / 学术 / 行业报告

### GitHub 信号（条件性）

如果 `sources` 中包含 GitHub 相关来源或搜索种子，或者用户的领域涉及开源项目：

- 检查 GitHub Trending 或使用 `gh search repos` + profile 中的搜索种子
- 如果最近 3 日内存在明显升温的相关项目，日报中纳入 1-2 条开源条目
- 重点记录：仓库名、定位、星数或热度信号、最近 push 时间、与用户相关的原因

如果用户领域与 GitHub / 开源无关，可跳过此步骤。

### 媒体源处理方式

媒体源优先从频道页建立候选池，再按需进入单篇文章正文。

## 五、推荐工具使用方式

可以使用以下任一方式采集：

- Agent Reach 已可用渠道
- 搜索工具
- fetch / reader 工具
- `scripts/save_raw_capture.py`

推荐流程：

1. 先运行 `agent-reach doctor`
2. 根据 doctor 结果，优先用 Agent Reach 已打通的上游工具建立 `index` 候选池，例如：
   - X/Twitter：`bird`
   - GitHub：`gh`
   - 网页正文：`curl https://r.jina.ai/URL`
   - Exa / 社交媒体：`mcporter`
   - RSS：RSS 读取器
3. 结合用户兴趣与候选质量做第一轮筛选
4. 对少量高价值条目再用 doctor 已确认可用的上游工具深抓正文
5. 将深抓结果保存为 `detail` 原始数据
6. 如果有任何来源改用备用工具，必须在原始采集记录或最终汇报里注明原因

GitHub 执行建议（仅在 profile 包含 GitHub 相关来源时执行）：

- 使用 `gh search repos` + profile 中的搜索种子查找相关项目
- 按最近活跃度和星数增速排序
- 重点记录：仓库名、定位、星数或热度信号、最近 push 时间、与用户相关的原因

如果使用 Agent Reach：

- 优先使用 `agent-reach doctor` 已标记为可用的上游工具
- 社交媒体和社区渠道可作为补充，不要替代一手源
- 抓到的是搜索摘要时，可在第二阶段再用 reader 深抓正文
- 如果 `agent-reach doctor` 显示某些渠道不可用，应记录不可用情况，再决定是否退回普通工具
- 最终汇报必须明确写出：
  - 是否运行了 `agent-reach doctor`
  - 本次实际使用了哪些 Agent Reach 上游工具
  - 哪些渠道因 doctor 显示不可用而被跳过

## 六、原始数据留存要求

这是强制要求：所有拉取到的资讯原始数据都必须留存，再进入筛选和加工。

默认输出：

- 第一阶段：`output/raw/{date}_index.txt`
- 第二阶段：`output/raw/{date}_detail.txt`

### 第一阶段建议保留

- 抓取时间
- 搜索词、频道页任务或来源页任务
- 来源名 / 来源类型
- 原始标题
- 原始链接
- 发布时间（如可获得）
- 简介 / 导语 / snippet

### 第二阶段建议保留

- 抓取时间
- 深挖任务
- 来源名 / 来源类型
- 原始标题
- 原始链接
- 语言
- 清洗后的正文、页面片段或发布说明
- 若抓取受阻，保留真实返回内容并标记异常状态

### 文本格式要求

- 每条记录用明显分隔线分开
- 元信息写成键值行
- 后面紧跟抓取到的原始文本
- 允许持续追加，不要求固定 JSON 结构

---

## 七、采集约束

1. 不要跳过第一阶段直接全文抓取所有文章
2. 不要只保留最后入选日报的条目，必须保留全部候选池
3. 原始文件内容应尽量贴近抓取结果本身，不要提前改写成日报文案
4. 第二阶段只深挖少量高价值条目，不要默认把所有候选全文抓下来
5. 如果页面存在明显验证码、反爬、挑战页，也应保留真实返回结果，并在后续筛选时降低其可用性

---

## 八、最小执行清单

进入筛选与加工前，至少确认：

- 已先运行 `agent-reach doctor`
- 已建立最近 3 日候选池
- 已优先覆盖 `sources.direct` 中配置的一手来源
- 已补充至少一部分社区 / 研究 / 多样性信号
- 如果 profile 包含 GitHub 相关来源，已检查 GitHub 热门 / 高活跃项目
- 已将候选池原始文本保存到 `index`
- 已对少量高价值条目完成 `detail` 深抓
- 已将深抓原始文本保存到 `detail`
- 如有退回普通工具，已记录退回原因与补齐来源
